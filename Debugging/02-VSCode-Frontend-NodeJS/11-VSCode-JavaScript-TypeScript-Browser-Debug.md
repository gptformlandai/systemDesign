# 11. VS Code: JavaScript, TypeScript, and Browser Debug

## Goal

Debug JavaScript and TypeScript in the browser using VS Code — source maps, Chrome DevTools attach, React component debugging, and TypeScript breakpoints.

---

## How Browser Debugging Works

```text
Browser (Chrome/Edge)
  -> Chrome DevTools Protocol (CDP)
  -> VS Code JavaScript Debug extension
  -> Your TypeScript/JSX source (via source maps)
```

VS Code connects to the browser's inspector using CDP, the same protocol Chrome DevTools uses.

---

## Chrome: Launch Configuration

```json
{
  "name": "Chrome: Launch",
  "type": "chrome",
  "request": "launch",
  "url": "http://localhost:3000",
  "webRoot": "${workspaceFolder}/src",
  "sourceMapPathOverrides": {
    "webpack:///src/*": "${webRoot}/*"
  }
}
```

VS Code opens a new Chrome window with debugging attached.

## Source Maps

Source maps link compiled/bundled JS back to your original TS/JSX source files.

### TypeScript tsconfig.json

```json
{
  "compilerOptions": {
    "sourceMap": true,
    "outDir": "./dist",
    "rootDir": "./src"
  }
}
```

### Webpack Source Map Config

```javascript
// webpack.config.js
module.exports = {
  devtool: 'source-map',         // production-grade, external .map files
  // or
  devtool: 'eval-source-map',    // development: fastest rebuild, inline maps
  // or
  devtool: 'cheap-module-source-map' // development: line-only, no column info
};
```

### sourceMapPathOverrides

When VS Code can't automatically resolve the source map path:

```json
"sourceMapPathOverrides": {
  "webpack:///src/*": "${workspaceFolder}/src/*",
  "webpack:///./src/*": "${workspaceFolder}/src/*",
  "meteor://💻app/*": "${workspaceFolder}/*"
}
```

## React Debugging

### React App (Create React App / Vite)

```json
{
  "name": "React: Launch Chrome",
  "type": "chrome",
  "request": "launch",
  "url": "http://localhost:3000",
  "webRoot": "${workspaceFolder}/src",
  "sourceMapPathOverrides": {
    "webpack:///src/*": "${webRoot}/*"
  }
}
```

With CRA, source maps are generated by default. Set breakpoints directly in `.jsx`/`.tsx` files.

### Debugging React Component Rendering

```tsx
function OrderList({ orders }: { orders: Order[] }) {
  // breakpoint here: inspect orders prop when component renders
  const filtered = orders.filter(o => o.status !== 'CANCELLED');
  // breakpoint here: inspect filtered result
  
  return (
    <ul>
      {filtered.map(order => (
        // breakpoint inside map callback: inspect each order
        <li key={order.id}>{order.id}: {order.total}</li>
      ))}
    </ul>
  );
}
```

### React DevTools + VS Code Together

```text
1. Install React DevTools Chrome extension.
2. Open VS Code Chrome launch debug session.
3. In Chrome: React DevTools shows component tree and props.
4. In VS Code: breakpoints in component functions show JS state.
   They complement each other.
```

## TypeScript Breakpoints

When source maps are correctly configured, you set breakpoints in `.ts` files:

```text
breakpoint on src/OrderService.ts:45
  -> VS Code finds the corresponding line in dist/OrderService.js via source map
  -> When JS engine hits that JS line, VS Code shows you TypeScript line 45
```

### Troubleshooting Unverified Breakpoints

Breakpoints appear as empty/grey circles (unverified) when source maps are not matched.

```text
Cause 1: outFiles not set — VS Code doesn't know where compiled JS is.
  Fix: add "outFiles": ["${workspaceFolder}/dist/**/*.js"]

Cause 2: sourceMapPathOverrides wrong prefix.
  Fix: check the source map paths in the browser Network tab -> Sources.

Cause 3: Source maps not generated.
  Fix: check tsconfig sourceMap: true and webpack devtool setting.
```

## Debugging webpack / Vite Applications

### Vite Dev Server (Fast)

```json
{
  "name": "Vite: Launch",
  "type": "chrome",
  "request": "launch",
  "url": "http://localhost:5173",
  "webRoot": "${workspaceFolder}/src",
  "sourceMapPathOverrides": {
    "/src/*": "${workspaceFolder}/src/*"
  }
}
```

Vite serves source files directly with native ES modules — source maps are one-to-one.

## Conditional Breakpoints In Browser Debug

```text
Right-click gutter in a .tsx/.js file -> Add Conditional Breakpoint
Condition: order.status === 'FAILED'
         user.role !== 'ADMIN'
         items.length === 0
```

These use the browser's JS engine to evaluate the condition.

---

## Logpoints In Browser Debug

```text
Right-click gutter -> Add Logpoint
Message: Order total: {order.total} status={order.status}
```

The output appears in the VS Code Debug Console and the browser's DevTools console.

Logpoints use `{}` not `${}` in VS Code.

## Interview Sound Bite

VS Code connects to Chrome via the Chrome DevTools Protocol (CDP). Source maps are the key to mapping breakpoints in TypeScript/JSX back to the compiled JS the browser runs. Set `sourceMap: true` in tsconfig and configure `outFiles` + `sourceMapPathOverrides` in launch.json. Unverified grey-circle breakpoints mean source map paths don't match. Vite is simpler: it serves original source files, so source maps are one-to-one. React components can be debugged by setting breakpoints directly in `.tsx` component functions — inspect props and state at render time.
