## Key Pair — The Core Concept

```
┌─────────────────────────────────────────────────────┐
│                  SSH KEY PAIR                        │
│                                                      │
│   PRIVATE KEY          PUBLIC KEY                   │
│   ┌──────────┐         ┌──────────┐                 │
│   │ id_ed25519│         │id_ed25519│                 │
│   │          │         │  .pub    │                 │
│   │ TOP SECRET│         │ SHAREABLE│                 │
│   └──────────┘         └──────────┘                 │
│   Stay on YOUR machine  → Go to server              │
│   Never share           → Lives in authorized_keys  │
└─────────────────────────────────────────────────────┘

Math relationship: What private key SIGNS, public key can VERIFY.
                   What public key ENCRYPTS, private key can DECRYPT.
```

---

## How Key-Based Authentication Works

```
STEP 1: You put your PUBLIC KEY on the server
  (into server's ~/.ssh/authorized_keys)

STEP 2: You connect with ssh
  Client: "I have a key, let me in"
  Server: "Prove it — sign this random challenge with your private key"

STEP 3: Client signs challenge with PRIVATE KEY
  Client → signed challenge → Server

STEP 4: Server verifies signature using YOUR PUBLIC KEY
  If signature valid → Access granted
  No password ever sent across the wire ✅
```

## Generating SSH Keys

```bash
# Generate Ed25519 key (RECOMMENDED — modern, fast, secure)
ssh-keygen -t ed25519 -C "your@email.com"

# Generate RSA 4096 key (older but widely compatible)
ssh-keygen -t rsa -b 4096 -C "your@email.com"

# With custom filename
ssh-keygen -t ed25519 -f ~/.ssh/github_key -C "github"

# With passphrase (prompted interactively — recommended)
# Press Enter to skip passphrase (less secure but convenient for CI/CD)
```

**Output:**
```
~/.ssh/id_ed25519      ← PRIVATE KEY (600 permissions)
~/.ssh/id_ed25519.pub  ← PUBLIC KEY  (644 permissions)
```

## Copy Public Key to Server

```bash
# Method 1: ssh-copy-id (easiest)
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@host

# Method 2: Manual copy
cat ~/.ssh/id_ed25519.pub | ssh user@host "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys"

# Method 3: If you already have shell access
echo "your-public-key-content" >> ~/.ssh/authorized_keys
```

## File Locations and Permissions

```
~/.ssh/                      ← Must be chmod 700
├── id_ed25519               ← Private key — chmod 600 (CRITICAL)
├── id_ed25519.pub           ← Public key — chmod 644
├── authorized_keys          ← chmod 600 (who can log in HERE)
├── known_hosts              ← Trusted server fingerprints
├── config                   ← SSH client config (chmod 600)
└── key.pem                  ← AWS PEM key — chmod 400

# Fix permissions if SSH refuses to connect:
chmod 700 ~/.ssh
chmod 600 ~/.ssh/id_ed25519
chmod 600 ~/.ssh/authorized_keys
```

## PEM Files — AWS and Cloud Keys

### What is a .pem file?
- PEM = **Privacy-Enhanced Mail** — a Base64-encoded key format
- It is just a **container format** for your private key
- Starts with: `-----BEGIN RSA PRIVATE KEY-----` or `-----BEGIN OPENSSH PRIVATE KEY-----`

### Why cloud providers (AWS, GCP, Azure) use PEM

```
Cloud scenario:
  - You create an EC2 instance
  - AWS generates a key pair
  - AWS keeps the PUBLIC key (installs it on your EC2)
  - You download the PRIVATE key as key.pem
  - That .pem IS your private key, just in PEM format

Usage:
  ssh -i ~/.ssh/my-aws-key.pem ec2-user@ec2-xxx.amazonaws.com
```

### PEM vs Normal SSH Key — Comparison

| Aspect | Normal SSH Key | PEM File |
|--------|---------------|----------|
| Format | OpenSSH format | PEM (PKCS#1 or PKCS#8) |
| Generated by | `ssh-keygen` | AWS console / openssl |
| Extension | `id_ed25519` / `id_rsa` | `.pem` |
| Content | Same private key | Same private key |
| Usage | `ssh user@host` | `ssh -i key.pem user@host` |
| Permission required | 600 | **400** (read-only) |

> **Key insight:** A .pem file IS a private key. The difference is just the encoding format, not the security concept.

### Convert PEM to OpenSSH format (if needed)

```bash
# PEM → OpenSSH
ssh-keygen -p -m PEM -f key.pem

# Extract public key from PEM
ssh-keygen -y -f key.pem > key.pub

# View key details
ssh-keygen -l -f key.pem
```

## Key Types — Which to Use

| Type | Algorithm | Key Size | Security | Speed | Recommendation |
|------|-----------|----------|----------|-------|----------------|
| Ed25519 | EdDSA (Curve25519) | Fixed 256-bit | Very High | Fastest | ✅ **Use this** |
| ECDSA | Elliptic Curve | 256/384/521 bit | High | Fast | Good |
| RSA | RSA | 2048/4096 bit | High (if 4096) | Slower | Legacy compat |
| DSA | DSA | 1024 bit | ❌ Weak | — | Never use |

## The ~/.ssh/known_hosts File

```bash
# View known hosts
cat ~/.ssh/known_hosts

# Remove a stale/changed server fingerprint
ssh-keygen -R hostname
ssh-keygen -R 192.168.1.10

# Manually verify server fingerprint
ssh-keyscan -H hostname >> ~/.ssh/known_hosts
```

**Warning:** If known_hosts shows a CHANGED fingerprint for a server you trust:
- Could be: server was rebuilt, IP reassigned
- Could be: **Man-in-the-Middle attack**
- → Verify out-of-band before accepting

## Mental Model: Key Distribution

```
DEVELOPER MACHINE          GITHUB / AWS / SERVER
~/.ssh/id_ed25519          ~/.ssh/authorized_keys
  (private key)      ←→     (public key stored here)

You PROVE you have the private key
Server VERIFIES using the public key
Never expose private key — the whole system breaks if you do
```
