# 05. Datadog APM: Tracing Concepts, SpanID, TraceID, Service Map, Sampling

## Goal

Understand distributed tracing from first principles: how a trace is formed, what spanID and traceID mean, how sampling works, and how to navigate traces in Datadog.

---

## What Is A Distributed Trace

A distributed trace captures the entire journey of a single request as it flows through multiple services.

```text
User request to API Gateway
  └── Span 1: api-gateway (HTTP /checkout)  [traceID=abc123, spanID=001, parentID=none]
        └── Span 2: orders-service (createOrder)  [traceID=abc123, spanID=002, parentID=001]
              └── Span 3: postgres (INSERT orders)  [traceID=abc123, spanID=003, parentID=002]
              └── Span 4: kafka (publish OrderCreated)  [traceID=abc123, spanID=004, parentID=002]
        └── Span 5: payments-service (charge)  [traceID=abc123, spanID=005, parentID=001]
              └── Span 6: stripe-api (HTTP POST)  [traceID=abc123, spanID=006, parentID=005]
```

All spans share the same traceID. The tree is built from parentID relationships.

---

## Core Concepts

| Concept | Definition |
|---|---|
| Trace | Complete lifecycle of one request across all services |
| Span | One unit of work: a service call, DB query, HTTP request, function call |
| traceID | 128-bit unique identifier shared by all spans in one trace |
| spanID | 64-bit unique identifier for one specific span |
| parentID | spanID of the calling span (forms the parent-child tree) |
| Root span | First span in a trace; no parentID |
| Entry span | First span in a service (service entry point) |
| Resource | The specific operation within a service (e.g., `GET /orders/{id}`) |
| Service | The application or process (e.g., `orders-service`) |
| Operation name | Type of span (e.g., `http.request`, `db.query`, `grpc.server`) |

## TraceID And SpanID Format

```text
traceID: 128-bit hex string
  "0af7651916cd43dd8448eb211c80319c"
  (or expressed as two 64-bit integers for W3C TraceContext)

spanID: 64-bit hex string (or integer)
  "b7ad6b7169203331"

parentID: same format as spanID
  "fa08e4c9427b8f88"
```

The traceID is generated by the root span. All downstream services propagate it via HTTP headers:

```text
# Datadog propagation headers.
x-datadog-trace-id: 7277407061855694839
x-datadog-parent-id: 3264694756489462988
x-datadog-sampling-priority: 1

# W3C TraceContext propagation headers.
traceparent: 00-0af7651916cd43dd8448eb211c80319c-b7ad6b7169203331-01
tracestate: dd=p:b7ad6b7169203331;s:1
```

## Sampling

Sampling controls what fraction of traces are kept. Datadog uses head-based sampling at the root span.

### Default Sampling (Adaptive)

The Datadog Agent automatically adjusts sampling rate per service to maintain ~50 traces/second per service. This is fully automatic.

### Rule-Based Sampling

Override default sampling with explicit rules:

```bash
# Environment variables for the tracer.
DD_TRACE_SAMPLE_RATE=0.1          # keep 10% of traces
DD_TRACE_SAMPLING_RULES='[{"service":"orders-service","sample_rate":1.0},{"sample_rate":0.05}]'
```

### Forced Sampling (Manual Override)

Always keep a specific trace regardless of sampling rules:

```java
// Java: force keep.
import datadog.trace.api.Trace;
import io.opentracing.util.GlobalTracer;

Span span = GlobalTracer.get().activeSpan();
span.setTag("manual.keep", true);
```

### Span Ingestion vs Retention

- **Ingestion**: spans sent to Datadog (controlled by agent-side sampling)
- **Retention**: spans stored after ingestion (controlled by retention filters in UI)

Errors and rare traces are always kept regardless of sample rate (Datadog built-in).

## Flame Graph View

The flame graph shows span hierarchy in time:

```text
Horizontal axis: time (span start and end)
Vertical axis: depth (span nesting)
Width of each bar: duration of that span

Red spans: errors
Dark bars: slow spans exceeding P99 threshold
```

---

## Service Map

APM → Service Map shows directed graph of service dependencies:

- Node = service
- Edge = calls between services
- Edge thickness = request volume
- Edge color = error rate (green → yellow → red)
- Node color = apdex score

## Interview Sound Bite

A distributed trace captures the complete request flow across all services. The traceID (128-bit) is unique to one request and shared across all spans. Each span has its own 64-bit spanID and records the parentID of its caller, forming a tree. Context propagation via HTTP headers (x-datadog-trace-id / W3C traceparent) is how traceID moves across service boundaries. Datadog uses head-based adaptive sampling, automatically targeting ~50 traces/second per service. Rule-based sampling lets you override this per service or globally.
