# 02. Document Model, BSON, and ObjectId

## Core Objects

| Concept | Meaning |
|---|---|
| Database | Logical namespace containing collections |
| Collection | Group of documents |
| Document | BSON object and atomic update boundary |
| Field | Key-value pair inside a document |
| Embedded document | Nested object inside a document |
| Array | Ordered list of values or documents |
| `_id` | Required primary key field with unique index |

## SQL Mapping

| SQL | MongoDB |
|---|---|
| Database | Database |
| Table | Collection |
| Row | Document |
| Column | Field |
| Join | Embed, reference, or `$lookup` |
| Primary key | `_id` |

## BSON

MongoDB stores BSON, not plain JSON.

Why BSON:

- supports dates, ObjectId, Decimal128, binary, and other types
- preserves type distinctions important for indexes
- is efficient for database traversal and storage

Common BSON types:

- string
- int32/int64
- double
- Decimal128
- boolean
- date
- ObjectId
- array
- embedded document
- null
- binary

## ObjectId

ObjectId is a common `_id` value generated by MongoDB drivers. At a high level, it includes timestamp-like, random/process, and counter components.

Implications:

- It is usually sortable by rough creation time.
- It can be generated client-side.
- It leaks approximate creation time.
- It is not a business ID.

## Document Size Limit

MongoDB documents are limited to 16 MB. This prevents one document from becoming an unbounded mini-database.

Avoid:

- all reviews inside one product
- all messages inside one conversation
- all audit events inside one user

Use:

- child collections
- bucket pattern
- subset pattern
- archives

## Schema Validation

MongoDB is schema-flexible, not schema-less.

```javascript
db.createCollection('users', {
  validator: {
    $jsonSchema: {
      bsonType: 'object',
      required: ['tenantId', 'email', 'createdAt'],
      properties: {
        tenantId: { bsonType: 'string' },
        email: { bsonType: 'string' },
        createdAt: { bsonType: 'date' }
      }
    }
  }
})
```

Use validation for critical collections so bad data does not silently become production debt.
